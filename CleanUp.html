Clean Architecture

Gente, acá les explico Clean Architecture de una forma clara: es una arquitectura que organiza el software para que las reglas importantes del negocio queden protegidas de frameworks, bases de datos, interfaces gráficas, APIs externas y demás detalles técnicos.

Concepto clave 🧠

Clean Architecture busca que el corazón del sistema no dependa de lo externo. El negocio debe poder entenderse, probarse y mantenerse sin estar amarrado a Spring, React, SQL, MongoDB, HTTP o cualquier tecnología que mañana puede cambiar.

1. ¿Qué es Clean Architecture?

Definición

Clean Architecture es una forma de organizar el código separando las reglas de negocio, los casos de uso y los detalles técnicos. La idea es que lo más importante del sistema quede en el centro, y lo externo quede en las capas de afuera.

El principio: las dependencias deben apuntar hacia adentro. Es decir, las capas externas pueden conocer las internas, pero las internas no deberían depender de las externas.

  • Las entidades representan reglas centrales del negocio.
  • Los casos de uso coordinan acciones de la aplicación.
  • Los controladores y bases de datos son detalles externos.
  • El dominio no debería saber qué framework se usa.
  • El sistema queda más fácil de probar, mantener y cambiar.

2. ¿Qué problema resuelve?

Problema típico

Cuando el negocio queda mezclado con controladores, SQL, frameworks o librerías externas, cambiar cualquier cosa se vuelve una vuelta pesada. Eso pasa cuando el código queda organizado por tecnología, pero no por responsabilidad real.

  • Las reglas de negocio terminan en controllers.
  • Los casos de uso quedan pegados a la base de datos.
  • Probar una regla requiere levantar toda la aplicación.
  • Cambiar framework o persistencia rompe más de lo necesario.
  • El código crece y nadie sabe dónde vive cada decisión importante.
Solución Clean

Vea pues: Clean Architecture separa el sistema por capas de responsabilidad. El negocio queda al centro, los casos de uso lo rodean y los detalles técnicos quedan por fuera.

  • Reglas de negocio protegidas.
  • Casos de uso claros y probables.
  • Infraestructura reemplazable.
  • Menor acoplamiento con frameworks.
  • Mayor claridad para equipos grandes.
Idea clave: Clean Architecture no es llenar el proyecto de carpetas. Es separar responsabilidades para que el negocio no quede secuestrado por la tecnología.

4. Capas principales de Clean Architecture

Entities

Son las reglas más importantes del negocio. No deberían depender de frameworks, bases de datos ni interfaces externas.

Use Cases

Representan lo que la aplicación puede hacer: crear tarea, registrar usuario, aprobar crédito o generar factura.

Interface Adapters

Adaptan datos entre el mundo externo y los casos de uso: controllers, presenters, gateways y mappers.

Frameworks & Drivers

Son detalles técnicos: Spring, Express, base de datos, UI, APIs externas, archivos o servicios cloud.

5. Regla de dependencia

La regla

Gente, esta es la regla más importante: las dependencias del código apuntan hacia el centro. Lo de afuera puede conocer lo de adentro, pero lo de adentro no debe depender de lo de afuera.

  • La entidad no conoce el controller.
  • El caso de uso no conoce la base de datos concreta.
  • El dominio no depende del framework.
  • La infraestructura implementa contratos definidos hacia adentro.
Ejemplo mental

Parce, si para cambiar de MySQL a MongoDB hay que tocar la regla de negocio, el diseño está demasiado acoplado. La entidad debería seguir igual, porque guardar datos es un detalle externo.

7. Organización Git base para Clean Architecture

Estructura base

Parce, vea es que en Git también se debe notar la arquitectura. Si todo queda mezclado, después nadie sabe qué es negocio, qué es caso de uso y qué es infraestructura.

Comandos sugeridos
git checkout develop git checkout -b feature/clean-task-usecase git add src/main/java/com/universidad/app/entities git add src/main/java/com/universidad/app/usecases git add src/main/java/com/universidad/app/adapters git commit -m "feat: implementar caso de uso de tareas con clean architecture" git push origin feature/clean-task-usecase

La idea es que el commit deje claro qué parte de la arquitectura cambió: entidad, caso de uso, adaptador o infraestructura.

Task.java

En este pedazo de código podemos ver la entidad. La tarea se protege a sí misma: no permite título vacío y define su estado inicial.

Ubicación en la arquitectura: Entities, ubicado en entities. Esta capa guarda reglas centrales del negocio y no depende de frameworks.

public class Task { private final String title; private final String description; private final String status; private Task(String title, String description) { if (title == null || title.isBlank()) { throw new IllegalArgumentException("El título de la tarea es obligatorio"); } this.title = title; this.description = description; this.status = "PENDING"; } public static Task create(String title, String description) { return new Task(title, description); } public String getTitle() { return title; } public String getDescription() { return description; } public String getStatus() { return status; } }

CreateTaskInput.java

Este archivo define lo que la aplicación puede hacer: crear una tarea. El controller llama este contrato sin conocer directamente la implementación.

Ubicación en la arquitectura: Use Cases / Input, ubicado en usecases/input. Representa el contrato de entrada del caso de uso.

public interface CreateTaskInput { TaskResponse createTask(CreateTaskRequest request); }

TaskRepositoryOutput.java

Este archivo define lo que el caso de uso necesita del mundo exterior: guardar una tarea. No dice si se guarda con JPA, MongoDB, archivo o memoria.

Ubicación en la arquitectura: Use Cases / Output, ubicado en usecases/output. Define una salida requerida por el caso de uso.

public interface TaskRepositoryOutput { Task save(Task task); }

CreateTaskInteractor.java

Este es el caso de uso. Recibe la solicitud, crea la entidad y manda guardar usando el contrato de salida. No sabe qué base de datos hay detrás, y eso está muy bien.

Ubicación en la arquitectura: Use Cases / Interactor, ubicado en usecases/interactor. Coordina entidades y salidas sin depender de infraestructura concreta.

public class CreateTaskInteractor implements CreateTaskInput { private final TaskRepositoryOutput taskRepositoryOutput; public CreateTaskInteractor(TaskRepositoryOutput taskRepositoryOutput) { this.taskRepositoryOutput = taskRepositoryOutput; } @Override public TaskResponse createTask(CreateTaskRequest request) { Task task = Task.create(request.title(), request.description()); Task savedTask = taskRepositoryOutput.save(task); return new TaskResponse( savedTask.getTitle(), savedTask.getDescription(), savedTask.getStatus() ); } }

TaskController.java

Este archivo recibe la petición externa. Gente, el controller no se pone a validar toda la lógica; traduce la entrada y llama el caso de uso.

Ubicación en la arquitectura: Adapters / Controllers, ubicado en adapters/controllers. Es un adaptador de entrada entre HTTP/UI y el caso de uso.

@RestController @RequestMapping("/tasks") public class TaskController { private final CreateTaskInput createTaskInput; public TaskController(CreateTaskInput createTaskInput) { this.createTaskInput = createTaskInput; } @PostMapping public ResponseEntity<TaskResponse> create(@RequestBody CreateTaskRequest request) { TaskResponse response = createTaskInput.createTask(request); return ResponseEntity.ok(response); } }

TaskPresenter.java

Este archivo ayuda a ordenar la respuesta para que la salida sea clara y consistente. Parce, una cosa es la entidad del negocio y otra la forma como se le responde al usuario.

Ubicación en la arquitectura: Adapters / Presenters, ubicado en adapters/presenters. Adapta datos de salida para presentación.

public class TaskPresenter { public TaskViewModel present(TaskResponse response) { return new TaskViewModel( response.title(), response.description(), "Estado actual: " + response.status() ); } }

TaskRepositoryGateway.java

Este archivo implementa el guardado real. Está por fuera del centro porque la base de datos es un detalle técnico, no la regla principal del negocio.

Ubicación en la arquitectura: Frameworks / Database, ubicado en frameworks/database. Implementa persistencia concreta.

public class TaskRepositoryGateway implements TaskRepositoryOutput { private final JpaTaskRepository jpaTaskRepository; public TaskRepositoryGateway(JpaTaskRepository jpaTaskRepository) { this.jpaTaskRepository = jpaTaskRepository; } @Override public Task save(Task task) { TaskEntity entity = new TaskEntity( task.getTitle(), task.getDescription(), task.getStatus() ); jpaTaskRepository.save(entity); return task; } }
Idea clave del ejemplo: la entidad protege la regla, el caso de uso coordina, el controller adapta la entrada y el gateway se encarga de la base de datos.

User.java

En este pedazo de código podemos ver la entidad de usuario. Esta clase valida reglas mínimas antes de permitir que un usuario exista dentro del sistema.

Ubicación en la arquitectura: Entities, ubicado en entities. Esta capa guarda reglas centrales del negocio y no depende de frameworks.

public class User { private final String name; private final String email; private final String password; private User(String name, String email, String password) { if (name == null || name.isBlank()) { throw new IllegalArgumentException("El nombre es obligatorio"); } if (email == null || !email.contains("@")) { throw new IllegalArgumentException("El correo no es válido"); } if (password == null || password.length() < 8) { throw new IllegalArgumentException("La contraseña debe tener mínimo 8 caracteres"); } this.name = name; this.email = email; this.password = password; } public static User register(String name, String email, String password) { return new User(name, email, password); } public String getName() { return name; } public String getEmail() { return email; } }

RegisterUserInput.java

Este archivo define el contrato de entrada. Es decir, lo que la aplicación permite hacer: registrar un usuario.

Ubicación en la arquitectura: Use Cases / Input, ubicado en usecases/input. Representa el contrato que usa el adaptador de entrada para llamar el caso de uso.

public interface RegisterUserInput { UserResponse registerUser(RegisterUserRequest request); }

UserRepositoryOutput.java

Este archivo define la salida hacia persistencia. El caso de uso necesita saber si un correo ya existe y también necesita guardar el usuario, pero no debe conocer la base de datos concreta.

Ubicación en la arquitectura: Use Cases / Output, ubicado en usecases/output. Define dependencias externas que el caso de uso necesita usar.

public interface UserRepositoryOutput { boolean existsByEmail(String email); User save(User user); }

RegisterUserInteractor.java

Este es el caso de uso. Valida si el correo ya existe, crea la entidad usuario y delega el guardado al contrato de salida.

Ubicación en la arquitectura: Use Cases / Interactor, ubicado en usecases/interactor. Coordina reglas del dominio y puertos de salida sin depender de infraestructura concreta.

public class RegisterUserInteractor implements RegisterUserInput { private final UserRepositoryOutput userRepositoryOutput; public RegisterUserInteractor(UserRepositoryOutput userRepositoryOutput) { this.userRepositoryOutput = userRepositoryOutput; } @Override public UserResponse registerUser(RegisterUserRequest request) { if (userRepositoryOutput.existsByEmail(request.email())) { throw new IllegalArgumentException("Ya existe un usuario con ese correo"); } User user = User.register( request.name(), request.email(), request.password() ); User savedUser = userRepositoryOutput.save(user); return new UserResponse( savedUser.getName(), savedUser.getEmail(), "USER_REGISTERED" ); } }

UserController.java

Este archivo recibe la petición HTTP. Gente, el controller no guarda usuarios directamente ni valida toda la regla de negocio. Traduce la entrada y llama el caso de uso.

Ubicación en la arquitectura: Adapters / Controllers, ubicado en adapters/controllers. Es un adaptador de entrada entre HTTP/UI y el caso de uso.

@RestController @RequestMapping("/users") public class UserController { private final RegisterUserInput registerUserInput; public UserController(RegisterUserInput registerUserInput) { this.registerUserInput = registerUserInput; } @PostMapping public ResponseEntity<UserResponse> register(@RequestBody RegisterUserRequest request) { UserResponse response = registerUserInput.registerUser(request); return ResponseEntity.ok(response); } }

UserPresenter.java

Este archivo prepara la respuesta que se entrega al usuario. Es importante porque no deberíamos devolver datos sensibles como la contraseña.

Ubicación en la arquitectura: Adapters / Presenters, ubicado en adapters/presenters. Adapta el resultado del caso de uso a una respuesta segura y entendible.

public class UserPresenter { public UserViewModel present(UserResponse response) { return new UserViewModel( response.name(), response.email(), "Usuario registrado correctamente" ); } }

UserRepositoryGateway.java

Este archivo implementa la persistencia real. Está en la capa externa porque usar JPA, SQL o cualquier motor de base de datos es un detalle técnico.

Ubicación en la arquitectura: Frameworks / Database, ubicado en frameworks/database. Implementa la persistencia concreta definida por el puerto de salida.

public class UserRepositoryGateway implements UserRepositoryOutput { private final JpaUserRepository jpaUserRepository; public UserRepositoryGateway(JpaUserRepository jpaUserRepository) { this.jpaUserRepository = jpaUserRepository; } @Override public boolean existsByEmail(String email) { return jpaUserRepository.existsByEmail(email); } @Override public User save(User user) { UserEntity entity = new UserEntity( user.getName(), user.getEmail() ); jpaUserRepository.save(entity); return user; } }
Idea clave del ejemplo: registrar un usuario no es solo guardar datos. Hay reglas de negocio, validaciones, persistencia y respuesta segura. Clean Architecture ayuda a que cada cosa quede en su sitio.

LoanApplication.java

En este pedazo de código podemos ver la entidad principal. Esta clase contiene reglas del negocio: valida ingresos, monto solicitado y decide si la solicitud puede aprobarse.

Ubicación en la arquitectura: Entities, ubicado en entities. Aquí viven reglas centrales y no debe haber dependencia de frameworks ni base de datos.

public class LoanApplication { private final String customerDocument; private final double monthlyIncome; private final double requestedAmount; private String status; private String reason; private LoanApplication(String customerDocument, double monthlyIncome, double requestedAmount) { if (customerDocument == null || customerDocument.isBlank()) { throw new IllegalArgumentException("El documento del cliente es obligatorio"); } if (monthlyIncome <= 0) { throw new IllegalArgumentException("El ingreso mensual debe ser mayor que cero"); } if (requestedAmount <= 0) { throw new IllegalArgumentException("El monto solicitado debe ser mayor que cero"); } this.customerDocument = customerDocument; this.monthlyIncome = monthlyIncome; this.requestedAmount = requestedAmount; this.status = "PENDING"; this.reason = "Solicitud pendiente de evaluación"; } public static LoanApplication create(String customerDocument, double monthlyIncome, double requestedAmount) { return new LoanApplication(customerDocument, monthlyIncome, requestedAmount); } public void evaluate(int riskScore) { double maxAllowedAmount = monthlyIncome * 8; if (riskScore < 650) { this.status = "REJECTED"; this.reason = "Puntaje de riesgo insuficiente"; return; } if (requestedAmount > maxAllowedAmount) { this.status = "REJECTED"; this.reason = "El monto supera la capacidad de pago estimada"; return; } this.status = "APPROVED"; this.reason = "Solicitud aprobada"; } public String getCustomerDocument() { return customerDocument; } public double getRequestedAmount() { return requestedAmount; } public String getStatus() { return status; } public String getReason() { return reason; } }

EvaluateLoanInteractor.java

Este es el caso de uso. Consulta el score de riesgo, crea la solicitud, ejecuta reglas de negocio, guarda el resultado y devuelve una respuesta.

Ubicación en la arquitectura: Use Cases / Interactor, ubicado en usecases/interactor. Coordina entidad, puerto externo y persistencia.

public class EvaluateLoanInteractor implements EvaluateLoanInput { private final RiskScoreOutput riskScoreOutput; private final LoanRepositoryOutput loanRepositoryOutput; public EvaluateLoanInteractor(RiskScoreOutput riskScoreOutput, LoanRepositoryOutput loanRepositoryOutput) { this.riskScoreOutput = riskScoreOutput; this.loanRepositoryOutput = loanRepositoryOutput; } @Override public LoanResponse evaluateLoan(LoanRequest request) { int riskScore = riskScoreOutput.getRiskScore(request.customerDocument()); LoanApplication loanApplication = LoanApplication.create( request.customerDocument(), request.monthlyIncome(), request.requestedAmount() ); loanApplication.evaluate(riskScore); LoanApplication savedLoan = loanRepositoryOutput.save(loanApplication); return new LoanResponse( savedLoan.getCustomerDocument(), savedLoan.getRequestedAmount(), savedLoan.getStatus(), savedLoan.getReason() ); } }

LoanController.java

Este archivo recibe la petición externa. El controller no calcula capacidad de pago, no decide aprobación y no consulta riesgo directamente. Solo adapta la entrada y llama el caso de uso.

Ubicación en la arquitectura: Adapters / Controllers, ubicado en adapters/controllers. Es el adaptador de entrada hacia el sistema.

@RestController @RequestMapping("/loans") public class LoanController { private final EvaluateLoanInput evaluateLoanInput; public LoanController(EvaluateLoanInput evaluateLoanInput) { this.evaluateLoanInput = evaluateLoanInput; } @PostMapping("/evaluate") public ResponseEntity<LoanResponse> evaluate(@RequestBody LoanRequest request) { LoanResponse response = evaluateLoanInput.evaluateLoan(request); return ResponseEntity.ok(response); } }

Este archivo prepara la respuesta para que sea entendible. Por ejemplo, puede convertir estados técnicos en mensajes claros para el usuario.

Ubicación en la arquitectura: Adapters / Presenters, ubicado en adapters/presenters. Adapta la salida del caso de uso.

public class LoanPresenter { public LoanViewModel present(LoanResponse response) { String message = response.status().equals("APPROVED") ? "Tu préstamo fue aprobado" : "Tu préstamo no fue aprobado: " + response.reason(); return new LoanViewModel( response.customerDocument(), response.requestedAmount(), response.status(), message ); } }

Este archivo implementa el guardado real en base de datos. Está afuera porque la base de datos es detalle técnico, no regla del negocio.

public class LoanRepositoryGateway implements LoanRepositoryOutput { private final JpaLoanRepository jpaLoanRepository; public LoanRepositoryGateway(JpaLoanRepository jpaLoanRepository) { this.jpaLoanRepository = jpaLoanRepository; } @Override public LoanApplication save(LoanApplication loanApplication) { LoanEntity entity = new LoanEntity( loanApplication.getCustomerDocument(), loanApplication.getRequestedAmount(), loanApplication.getStatus(), loanApplication.getReason() ); jpaLoanRepository.save(entity); return loanApplication; } }

RiskScoreGateway.java

Este archivo conecta con un proveedor externo de score de riesgo. El caso de uso no sabe si se consulta una API, un servicio interno o una librería. Solo usa el contrato.

Ubicación en la arquitectura: Frameworks / External, ubicado en frameworks/external. Implementa integración con sistemas externos.

public class RiskScoreGateway implements RiskScoreOutput { private final RiskScoreClient riskScoreClient; public RiskScoreGateway(RiskScoreClient riskScoreClient) { this.riskScoreClient = riskScoreClient; } @Override public int getRiskScore(String customerDocument) { RiskScoreResponse response = riskScoreClient.findScore(customerDocument); return response.score(); } }
Idea clave del ejemplo: la decisión de aprobación vive en la entidad y el caso de uso coordina. La base de datos y el proveedor de riesgo son detalles externos.

11. Testing en Clean Architecture

Entities

Gente, acá se prueban las reglas más importantes del negocio: validaciones, estados, decisiones y restricciones.

Use Cases

Se prueba el flujo de aplicación usando repositorios simulados o fakes. Así no toca levantar base de datos para validar una regla.

Adapters

Se revisa que controllers, presenters y gateways traduzcan bien datos entre el mundo externo y el caso de uso.

Frameworks

Se prueban integraciones reales con base de datos, APIs externas o frameworks cuando ya toca validar infraestructura.

Ejemplo de prueba: LoanApplicationTest.java

En este pedazo de código podemos ver una prueba sobre la entidad. No levanta Spring, no toca base de datos y no consulta servicios externos. Va directo a la regla.

Ubicación en la arquitectura: Entity Unit Test, ubicado en tests/entities. Valida reglas centrales del negocio sin infraestructura.

import org.junit.jupiter.api.Test; import static org.junit.jupiter.api.Assertions.assertEquals; public class LoanApplicationTest { @Test void shouldRejectLoanWhenRiskScoreIsLow() { LoanApplication loan = LoanApplication.create( "100200300", 3000000, 5000000 ); loan.evaluate(580); assertEquals("REJECTED", loan.getStatus()); assertEquals("Puntaje de riesgo insuficiente", loan.getReason()); } }

Ejemplo de prueba: RegisterUserInteractorTest.java

Este ejemplo prueba un caso de uso. Parce, acá no necesitamos una base de datos real: usamos un repositorio falso para validar el flujo.

Ubicación en la arquitectura: Use Case Test, ubicado en tests/usecases. Valida la coordinación del caso de uso con dependencias simuladas.

import org.junit.jupiter.api.Test; import static org.junit.jupiter.api.Assertions.assertThrows; public class RegisterUserInteractorTest { @Test void shouldRejectUserWhenEmailAlreadyExists() { UserRepositoryOutput fakeRepository = new FakeUserRepository(true); RegisterUserInput registerUser = new RegisterUserInteractor(fakeRepository); RegisterUserRequest request = new RegisterUserRequest( "Ana Gómez", "ana@example.com", "password123" ); assertThrows( IllegalArgumentException.class, () -> registerUser.registerUser(request) ); } }
Idea clave: Clean Architecture facilita probar desde adentro hacia afuera: primero entidades, luego casos de uso, después adaptadores y por último infraestructura.

12. Ventajas y riesgos

Ventajas
  • Negocio protegido: las reglas importantes no dependen del framework.
  • Testing más limpio: se pueden probar entidades y casos de uso sin infraestructura.
  • Menor acoplamiento: base de datos, web y APIs externas quedan como detalles.
  • Mantenibilidad: el equipo entiende mejor dónde vive cada responsabilidad.
  • Flexibilidad tecnológica: cambiar framework o persistencia impacta menos el centro.
Riesgos
  • Más estructura: aparecen más interfaces, carpetas y clases.
  • Curva de aprendizaje: si el equipo no entiende la regla de dependencia, puede enredarse.
  • Sobrearquitectura: para CRUD pequeños puede ser demasiado.
  • Falsa limpieza: crear carpetas bonitas no garantiza buen diseño.
  • Mapeos excesivos: DTOs, entidades y modelos mal manejados pueden volver el código pesado.
🧼🧱

Meme: “le puse carpetas entities, usecases y adapters, entonces ya es Clean”. No, gente. Si las dependencias siguen apuntando para cualquier lado, eso solo es desorden con nombres bonitos.

13. ¿Cuándo usar Clean Architecture?

Úsela cuando...
  • El sistema tiene reglas de negocio importantes.
  • Se espera que el proyecto crezca y sea mantenido por varios desarrolladores.
  • Se necesita probar lógica sin depender de base de datos, servidor o framework.
  • Hay integraciones externas que pueden cambiar con el tiempo.
  • El equipo quiere separar dominio, casos de uso, adaptadores e infraestructura.
  • Se trabaja en sistemas empresariales, financieros, académicos, logísticos o de salud.

14. ¿Cuándo NO usar Clean Architecture?

No siempre hace falta

Gente, Clean Architecture no es para meterla a la fuerza en todo. Si la aplicación es muy pequeña o solo tiene CRUD sin reglas relevantes, puede terminar siendo más ceremonia que solución.

  • Aplicaciones demasiado pequeñas o prototipos rápidos.
  • CRUD simple sin reglas de negocio importantes.
  • Equipos que todavía no entienden separación de responsabilidades.
  • Proyectos donde el costo de estructura supera el beneficio.
  • Casos donde se necesita entregar una prueba de concepto muy rápida.
Señales de alerta
  • Hay interfaces para todo, aunque solo exista una implementación trivial.
  • El equipo copia la estructura sin entender la regla de dependencia.
  • El controller sigue teniendo reglas de negocio.
  • La entidad depende de JPA, Spring, HTTP o JSON.
  • Los casos de uso parecen servicios gigantes con muchas responsabilidades.
Idea clave: Clean Architecture debe resolver un problema de mantenibilidad, pruebas y acoplamiento. Si no hay ese problema, puede ser demasiado para el caso.

15. Comparación: Clean vs Hexagonal vs Capas

Criterio Arquitectura por capas Arquitectura Hexagonal Clean Architecture
Centro del diseño Capas técnicas: controller, service, repository. Dominio y puertos/adaptadores. Entidades, casos de uso y regla de dependencia.
Dependencias Pueden ir de arriba hacia abajo y acoplarse al framework. El dominio se comunica mediante puertos. Las dependencias apuntan hacia el centro.
Dominio A veces queda mezclado en servicios. Queda protegido en el centro. Queda como capa interna junto a entidades y reglas centrales.
Testing Puede requerir más infraestructura. Permite probar dominio y casos de uso con adaptadores simulados. Facilita pruebas por capas internas y externas.
Uso recomendado Aplicaciones sencillas o medianas. Sistemas donde se quieren aislar entradas y salidas. Sistemas con reglas de negocio fuertes y necesidad de mantenibilidad.
Riesgo común Services gigantes llenos de lógica. Crear muchos puertos sin criterio. Sobrearquitectura y exceso de mapeos.

16. Relación con DDD, Testing y Microservicios

DDD

Clean Architecture se lleva muy bien con DDD porque ayuda a proteger el dominio. Las entidades y reglas importantes pueden vivir en el centro sin depender de infraestructura.

Testing

Como el negocio queda aislado, se pueden crear pruebas rápidas y claras. Primero se validan reglas, luego casos de uso y después integraciones externas.

Microservicios

Un microservicio puede estar construido internamente con Clean Architecture. Microservicios separa el sistema en servicios; Clean ordena el interior de cada servicio.

17. Cierre

Gente, si se llevan una sola idea, que sea esta: Clean Architecture protege las reglas importantes del negocio y organiza las dependencias hacia el centro.

El controller, la base de datos, el framework y las APIs externas son importantes, claro que sí, pero no deberían mandar sobre el negocio. Son detalles externos.

Cuando Clean Architecture se aplica bien, el sistema queda más fácil de probar, mantener, evolucionar y explicar. Cuando se aplica mal, solo quedan muchas carpetas, muchas interfaces y el mismo desorden de siempre.
